# Long-Running Agent Scaffold

A portable, project-agnostic harness for multi-session AI coding agents. Drop it into any project, run the initializer, and get a self-reflecting agent team that works across unlimited context windows.

Based on [Anthropic's long-running agent research](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents) and [OpenAI's harness engineering](https://openai.com/zh-Hans-CN/index/harness-engineering/).

## The Problem

| Failure Mode | Cause |
|---|---|
| Agent tries to build everything at once | No incremental task structure |
| New session has no idea what happened before | No persistent state across context windows |
| Agent declares "done" without proof | No verification protocol |
| Same mistake happens in session 3 and session 7 | No reflection loop — errors aren't turned into guards |
| Switching to a new project means rewriting all agent config | Domain knowledge hardcoded into prompts |

## The Solution: Three Layers

```
┌─────────────────────────────────────────────────────────┐
│  HARNESS (generic, portable)                            │
│  Agents · Session protocols · State management          │
├─────────────────────────────────────────────────────────┤
│  DOMAIN ADAPTER (pluggable, one file per project)       │
│  Build · Test · Deploy · Verify · Authenticate          │
├─────────────────────────────────────────────────────────┤
│  PROJECT STATE (runtime artifacts)                      │
│  state.json · task_plan.json · notes.md                 │
└─────────────────────────────────────────────────────────┘
```

The **harness** is 100% generic. The **adapter** is the only project-specific file — and it's auto-generated by scanning your codebase.

## Quick Start

### 1. Install the scaffold

```bash
cp -r .claude/ /path/to/your-project/.claude/
```

### 2. Initialize (scans your project automatically)

```
Use the initializer agent to set up this project
```

The initializer will:
1. **Scan** your codebase — package.json, CI configs, test configs, Dockerfiles, git history, source conventions
2. **Generate** `domain/adapter.md` with real build/test/deploy/verify commands
3. **Create** `state.json`, `task_plan.json`, `notes.md`, `init.sh`
4. **Verify** the scaffold works (runs build + test)
5. **Commit** everything

### 3. Code (every subsequent session)

```
Use the coder agent to make progress
```

The coder will:
1. **Orient** — read `state.json` (one file, instant context), check health
2. **Choose** the next pending task from `task_plan.json`
3. **Implement** with TDD
4. **Observe** — run the adapter's verify protocol after deploys
5. **Reflect** — assess outcome, detect patterns, create guards, update knowledge
6. **Commit** and update all state files
7. **Repeat** until context limit or all tasks done

### 4. Garden (periodically)

```
Use the gardener agent to clean up
```

Detects stale docs, pattern drift, flaky tests, and state file inconsistencies. Opens cleanup commits.

## Agents

| Agent | When | What It Does |
|---|---|---|
| **initializer** | Once per project | Scans codebase → generates adapter + state + plan |
| **coder** | Every session | Orient → implement → observe → reflect → commit |
| **gardener** | Every 10+ sessions | Detect entropy → clean docs → fix drift → update quality score |

## Session Lifecycle

```
SESSION START
  ├─ Read state.json (instant context)
  ├─ Read task_plan.json + notes.md + git log
  ├─ Read domain/adapter.md (how to build/test/deploy)
  ├─ Run init.sh + smoke test
  │   └─ If broken → FIX FIRST
  │
IMPLEMENT (one task)
  ├─ TDD: test → implement → verify
  ├─ Commit with descriptive message
  │
OBSERVE (if deployed)
  ├─ Run adapter's verify methods
  ├─ Check success/failure markers
  │   └─ If fails → FIX, don't proceed
  │
REFLECT
  ├─ Assess outcome (success/fail/partial)
  ├─ Pattern recognition (recurring issue?)
  ├─ Create guard if recurring (ADR, test, gotcha)
  ├─ Update domain knowledge if new learning
  ├─ Adjust plan if needed
  │
UPDATE STATE
  ├─ state.json ← session summary
  ├─ task_plan.json ← task status
  ├─ notes.md ← timestamped log
  │
REPEAT or STOP (clean state)
```

## File Structure

```
.claude/
├── CLAUDE.md                     # ~60 lines: mission, rules, file map
├── agents/
│   ├── initializer.md            # Auto-scan → generate scaffold
│   ├── coder.md                  # Orient → implement → observe → reflect
│   └── gardener.md               # Periodic quality maintenance
├── harness/
│   ├── session-start.md          # 6-step startup protocol
│   ├── session-reflect.md        # 6-step post-task reflection
│   ├── session-observe.md        # 6-step post-deploy verification
│   └── progress-tracker.md       # State file management
├── domain/
│   ├── adapter.md                # ★ THE PLUGGABLE PART (auto-generated)
│   └── knowledge/                # Domain docs (populated during work)
├── templates/
│   ├── state.json                # Machine-readable state template
│   ├── task_plan.json            # Structured task list template
│   └── adapter-example.md        # 3 examples: Nx/CDK, Next.js, FastAPI
└── docs/
    ├── decisions/                # ADRs from reflection loop
    └── evidence/                 # Verification artifacts
```

### Runtime Files (created by initializer, managed by coder)

| File | Format | Purpose |
|---|---|---|
| `state.json` | JSON | Machine-readable project state — single-file session context |
| `task_plan.json` | JSON | Structured feature/task list with status per task |
| `notes.md` | Markdown | Human-readable session logs, decisions, errors |
| `init.sh` | Shell | Idempotent dev environment setup |

## The Adapter Pattern

The adapter is the **only file you customize per project**. The initializer generates it automatically by scanning your codebase. It answers six questions:

| Section | What It Tells the Agent |
|---|---|
| **Build** | How to compile/bundle this project |
| **Test** | How to run unit, integration, E2E tests |
| **Deploy** | How to deploy and to which environments |
| **Verify** | How to confirm a deploy actually works |
| **Authenticate** | How to get tokens/keys for API calls |
| **Conventions** | Naming, commit messages, code patterns |

See `templates/adapter-example.md` for three full examples covering Nx monorepo with CDK, Next.js on Vercel, and Python FastAPI with Docker.

## Why JSON for State?

Anthropic's research found that models accidentally corrupt markdown files but respect JSON structure. Using JSON for `state.json` and `task_plan.json` means:
- The agent can't accidentally delete tasks by editing markdown checkboxes
- Session startup is one `read state.json` instead of parsing three files
- Task status changes are precise field updates, not text substitution

`notes.md` stays markdown because it's an append-only human-readable log.

## Design Principles

These come directly from the two papers:

| Principle | Source | Implementation |
|---|---|---|
| One task at a time | Anthropic | Coder agent picks ONE pending task per cycle |
| Fix broken state first | Anthropic | Session-start smoke tests before new work |
| Feature list in JSON | Anthropic | `task_plan.json` with structured status fields |
| Progress file + git history | Anthropic | `state.json` + `notes.md` + `git log` |
| Repo is the system of record | OpenAI | All knowledge in `.claude/domain/`, not external tools |
| AGENTS.md as table of contents | OpenAI | `CLAUDE.md` is ~60 lines, points to deeper files |
| Progressive disclosure | OpenAI | Agent reads `state.json` first, expands as needed |
| Encode taste into tools | OpenAI | Reflection loop creates guards from recurring patterns |
| Garbage collection | OpenAI | Gardener agent detects and fixes entropy |
| Make the app readable | OpenAI | Observe protocol wires verify/logs into the lifecycle |
